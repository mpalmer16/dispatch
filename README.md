# Dispatch

Dispatch is a small Rust-based distributed systems playground.

The goal of this project is to explore how real backend systems behave when multiple services, retries, and asynchronous workflows interact. Rather than building a large application, Dispatch focuses on implementing small but realistic patterns such as idempotent APIs, event-driven workflows, and reliable message processing.

The entire system runs locally using Docker and is designed to be easy to experiment with.

---

## Current Features

Order service implemented in Rust using `axum` and `sqlx`.

PostgreSQL persistence with migrations.

Idempotent order creation using an `Idempotency-Key` header.

Conflict detection when the same idempotency key is reused with a different payload.

Integration tests against a dedicated test database.

Structured service setup with shared application state and clean request handlers.

---

## API

### Create Order

`POST /orders`

Headers

```
Content-Type: application/json
Idempotency-Key: optional
```

Body

```json
{
  "customer_id": "customer-123",
  "total_cents": 4999
}
```

Responses

**201 Created**

Returns the created order.

**200 OK**

Returned when the same idempotency key is reused with the same payload.

**409 Conflict**

Returned when the same idempotency key is reused with a different payload.

---

### Get Order

`GET /orders/{id}`

Returns the order if it exists.

**200 OK**

Returns the order.

**404 Not Found**

Order does not exist.

## Running Locally

Start PostgreSQL:

```bash
docker compose up -d
```

Run the service:

```bash
cargo run -p order-service
```

Health check:

```bash
curl http://localhost:3000/health
```

Create an order:

```bash
curl -X POST http://localhost:3000/orders \
  -H "content-type: application/json" \
  -H "Idempotency-Key: example-123" \
  -d '{"customer_id":"customer-1","total_cents":4999}'
```

## Running Tests

Tests use a dedicated database.

```
TEST_DATABASE_URL=postgres://dispatch:dispatch@localhost:5432/dispatch_test
```

Run tests:

```bash
cargo test -p order-service
```

The test database is truncated between test runs to ensure isolation.

## Architecture

```

│   ├── order-service

├── migrations
├── docker-compose.yml
```

Dispatch exists to explore those interactions in a small, understandable system built from the ground up.

The goal is not to build a production platform, but to better understand the patterns used in real distributed systems.

## Development Notes

### Idempotency

Order creation supports idempotent requests using an `Idempotency-Key` header.

This allows clients to safely retry requests if they are unsure whether a previous request succeeded.

Behavior:

| Scenario | Result |
|--------|-------|
| Same key + same payload | Existing order returned |
| Same key + different payload | `409 Conflict` |
| No key | Each request creates a new order |

This mirrors behavior used by real-world APIs such as payment providers.

---

### Test Database Isolation

Integration tests run against a dedicated test database (`dispatch_test`).

The test harness truncates the database between tests to ensure isolation.

This prevents tests from interacting with development data and ensures deterministic results.

---

### Integration Testing Strategy

Tests are written as API-level integration tests rather than unit tests.

Each test:

1. Builds the application router
2. Sends HTTP requests directly to the service
3. Verifies responses and database behavior

This approach ensures that the entire request path is tested:

```
HTTP request
   ↓
Router
   ↓
Handler
   ↓
Database
   ↓
Response serialization
```

This style of testing catches real behavioral issues that unit tests might miss.

---

### Why Rust?

Rust is used for this project because it encourages explicit modeling of system behavior and failure cases.

For distributed systems work in particular, Rust provides:

- strong type guarantees
- clear async concurrency with `tokio`
- predictable performance
- compile-time safety around many common runtime errors

The goal is not simply to build an application in Rust, but to explore systems patterns using Rust as the implementation language.
